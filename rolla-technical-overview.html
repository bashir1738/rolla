Rolla

A trustless, on-chain rotating savings platform — from smart contract to mobile app. This document covers the full technical stack, every SDK and package used, and the rationale for building on a public blockchain.

Ethereum Sepolia Solidity 0.8 React Native / Expo Viem · Wagmi Magic SDK Uniswap V3
Network — Ethereum Sepolia (Chain ID 11155111) Date — June 2026 Author — Gbadamosi Bashir Olamide

1. Executive Summary

Rolla is a DeFi-native mobile savings platform built on the ajo / esusu / adashe rotating savings model practiced across West Africa and the diaspora. Members pool a fixed contribution each round; the full pot is released to one member per round, rotating until every member has received once.

Three smart contracts deployed on Ethereum Sepolia form the trustless core. A React Native / Expo mobile app (Android) and a Next.js marketing site sit on top. All financial logic — pot custody, payouts, yield tiers, and on-chain identity — lives in immutable Solidity code, not in a company-controlled database.

2. Smart Contracts

All contracts are written in Solidity ^0.8.20, compiled and tested with Hardhat, and deployed via Hardhat Ignition. They share no proxy pattern — code is immutable at the addresses below.

2.1 AjoCircle — Rotating Savings Circles

This is the primary product contract. It manages the full lifecycle of a savings circle: recruiting members, collecting contributions, and releasing the pot.

Key Design Decisions

  • Any EVM token as input. Members call contribute() with any token (or native ETH). The contract atomically swaps it to USDC via Uniswap V3 ExactInputSingle before crediting the pool. Excess USDC above the contribution amount is returned to the sender.
  • Any EVM token as payout output. The recipient calls claimPayout() specifying their preferred tokenOut. USDC is swapped back via Uniswap V3 before delivery.
  • Deterministic 6-digit IDs. Circle IDs are derived from keccak256(prevrandao, sender, count) modulo 900000, making them memorable and shareable (e.g., "join circle 483291").
  • Auto-activation. A circle transitions from Recruiting to Active the moment the last seat is filled — no admin step required.
  • Reentrancy guard. All external state-changing functions use a custom ReentrancyGuard.

2.2 RollaVault — Tiered Savings Vaults

Provides individual yield-bearing savings vaults. Depositors lock USDC for a chosen tier and earn projected interest proportional to elapsed time and the tier's APR.

TierLock PeriodAPR (bps)Min Deposit
Flex (0)No lock450 bps (4.5%)$10 USDC
Growth (1)90 days920 bps (9.2%)$100 USDC
Power (2)365 days1,480 bps (14.8%)$500 USDC

The deposit flow mirrors AjoCircle: any input token is swapped to USDC via Uniswap V3. On claim, USDC is optionally swapped to the user's preferred output token. The aTokenShares field on the Vault struct is reserved for a future Aave V3 integration that will compound the principal on mainnet.

2.3 UsernameRegistry — On-Chain Identity

A lightweight registry mapping wallet addresses to unique human-readable names. No ENS dependency — the registry is self-contained.

  • 1–32 characters: a–z, A–Z, 0–9, -, _
  • Case-insensitive uniqueness — stored and compared in lowercase
  • Free to claim — user pays gas only
  • One name per address; claiming a new one auto-releases the previous
  • 24-hour cooldown between claims to prevent namespace cycling

3. Mobile Application

The mobile app is an Expo (React Native) application targeting Android, built with the Expo Dev Client (not Expo Go) due to the native Magic SDK modules. It is distributed as an APK via Expo Application Services (EAS).

3.1 Authentication — Magic Link SDK

Users authenticate with their email address via Magic's Email OTP flow. Magic provisions a non-custodial embedded wallet behind the scenes — the user never sees a seed phrase. The wallet is injected into wagmi via a custom MagicConnector.

A PIN system layered on top of Magic (stored in expo-secure-store) adds a fast re-auth gate without requiring the user to re-enter their email every session.

3.2 Transaction Layer — sendTx Utility

Rather than using wagmi's writeContractAsync (which has first-click failures when Magic's WebView isn't fully warm), a custom sendTx() utility:

  1. Fetches the wallet address directly via magic.rpcProvider.request({ method: 'eth_accounts' })
  2. Uses viem's encodeFunctionData to ABI-encode the calldata
  3. Estimates gas via the public Infura/Sepolia RPC, adds a 30% buffer
  4. Signs and broadcasts via eth_sendTransaction through Magic's relay
  5. Waits for receipt via viem's publicClient.waitForTransactionReceipt()

All network calls are wrapped in withTimeout() to surface failures quickly rather than hanging indefinitely.

3.3 On-Chain Data Reading — wagmi + viem

Contract state is read using wagmi's useReadContract and useReadContracts hooks (batch multicall). All reads poll every 30 seconds to stay in sync with chain state.

3.5 Quote Engine

The useQuote() hook queries the Uniswap V3 Quoter contract (0xEd1f6473…) off-chain before any swap-based contribution or deposit to preview the expected USDC output. This prevents users from accidentally falling below the minimum contribution threshold.

3.6 Push Notifications

expo-notifications and expo-task-manager with expo-background-fetch are used to alert users when a circle payout is pending, when a new round starts, or when a vault matures.

4. Marketing Website (Next.js)

A statically rendered Next.js 16 site (App Router) deployed to Vercel. It serves as the product landing page and includes a live ClaimUsername widget that connects to the UsernameRegistry contract via viem — the only on-chain interaction on the web frontend.

  • App Router with server components by default
  • Tailwind CSS v4 for styling
  • No external auth — read-only contract calls and username registration only
  • GetApp modal deep-links to the Expo EAS APK build for Android download

6. Why Rolla Must Be On-Chain

The ajo/esusu savings model is built entirely on trust between participants. Traditional circles fail for one reason: the organizer holds the money. They can disappear, delay payouts, or simply be unreachable. Moving to a centralized app solves the disappearance problem but replaces the organizer risk with company risk — the platform can freeze accounts, go bankrupt, or geofence users. On-chain execution eliminates both risks at the source.

01

No Custodial Risk

Funds are held in the smart contract, not in a company bank account. Rolla the company cannot move, freeze, or seize funds. The rescueETH() admin function only recovers accidentally-sent ETH — it cannot touch USDC inside any circle or vault.

02

Trustless Payout Order

Payout positions are set permanently when members join and recorded on-chain. No organizer can skip someone, change the order, or grant themselves an extra round. The contract enforces rotation deterministically.

03

Permissionless & Global

Anyone with a wallet can create or join a circle — no KYC, no bank account, no geographic restriction. This is critical for diaspora communities sending value back home and for users in countries with limited banking infrastructure.

04

Transparent Accounting

Every contribution, payout, and membership change is a public transaction on Sepolia. Any member can independently verify the pool balance and round status without trusting a dashboard — they can query the contract directly.

05

Atomic Multi-Token Swaps

Because Uniswap V3 is also on-chain, contributions and payouts can accept any EVM token in the same atomic transaction. This is impossible to replicate safely with off-chain systems without introducing counterparty risk at the swap layer.

06

Composable with DeFi

Circle pools and vault deposits can be routed into Aave V3 in a future upgrade to earn yield while idle. This composability — using one protocol's liquidity inside another — is unique to on-chain finance and impossible with closed banking APIs.

07

Censorship Resistance

Once deployed, the contracts run forever on Ethereum's validator network. No government, bank, or company can shut down the core savings logic. Even if the Rolla app disappears, members can interact with the contracts directly via Etherscan or any EVM wallet.

08

On-Chain Identity

Usernames registered in UsernameRegistry follow the wallet — not the app. If a user switches to a different Rolla client in the future, their identity, history, and circle memberships are all intact on-chain.

Summary

A centralised savings app can automate the ledger. Only an on-chain app can eliminate the need to trust the ledger keeper. Rolla's value proposition — saving in a circle with people you may only partially trust, across borders, without a common bank — only works if the custody, the order, and the accounting are enforced by math, not by humans.
